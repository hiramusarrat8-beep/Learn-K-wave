% ── Step 1: define grid dimensions ──────────────
Nx = 64; Ny = 64; Nz = 64; % grid points
Grid point count
Each axis has 64 grid points. Total: 64³ = 262,144 voxels. At single precision (4 bytes), the medium matrices alone need ~3 MB. k-Wave uses ~21 matrices internally → ~60 MB total. This fits in MATLAB Online without a problem.
dx = 0.5e-3; dy = 0.5e-3; dz = 0.5e-3; % [m]
Grid spacing from PPW rule
For a 500 kHz transducer in water (c=1500 m/s): λ = c/f = 3mm. Use 6 PPW for accuracy: dx = λ/6 = 0.5 mm. The grid is isotropic (dx=dy=dz), which is correct for a bowl transducer.
dx = c_water / (PPW × f0) = 1500 / (6 × 500,000) = 0.0005 m
% ── The key change from 2D ──────────────────────
kgrid = kWaveGrid(Nx, dx, Ny, dy, Nz, dz);
🔑 The only structural difference from 2D
In Level 2 you wrote kWaveGrid(Nx,dx,Ny,dy) — 4 arguments. Here you add one more pair: Nz,dz. k-Wave detects the number of argument pairs and runs the 3D solver automatically. That's it.
% ── medium: now Nx×Ny×Nz matrices ──────────────
medium.sound_speed = 1500; % scalar = uniform water
Scalar vs matrix — same rule as 2D
Scalar means "same value everywhere in the 3D grid." In Level 4 you'll replace this with a real Nx×Ny×Nz matrix from CT data. For now, scalar is fine — pure water, no skull.
medium.density = 1000; % scalar = uniform water
kgrid.makeTime(medium.sound_speed, 0.3); % CFL=0.3
CFL rule — identical to 2D
The Courant-Friedrichs-Lewy condition works the same in 3D. makeTime(c, 0.3) sets dt so that sound travels at most 30% of one grid cell per time step. In 3D, always pass c_max — if you add skull later, pass 2900, not 1500.

makeBowl — the 3D transducer
Replaces makeArc from Level 2. Same idea, real 3D geometry.
💡
Why not makeArc in 3D? In 2D, makeArc draws a curved line — which represents an infinite cylindrical source extruded into the z-direction. In 3D, you want an actual spherical cap (bowl). makeBowl gives you the real thing.

% Bowl position: back-wall face, centred in y and z
bowl_pos = [1, Ny/2+1, Nz/2+1]; % [grid pts]
Bowl position = centre of rear surface
bowl_pos is the grid-index location of the centre of the back face of the transducer — where the cable plugs in. x=1 means it sits at the very edge of the grid. y and z are centred. All three values are grid indices (integers), not physical metres.
focus_pos = [Nx/2, Ny/2+1, Nz/2+1]; % [grid pts]
focus_pos = geometric intent, NOT guaranteed
This tells makeBowl which direction the bowl faces — it sets the geometry of the spherical cap. In water, the focus will land here. Once you add skull (Level 4), refraction shifts the actual focal spot away from this point. Never move bowl_pos to fix a shifted focus.
radius = round(60e-3 / dx); % radius of curvature
Radius of curvature → focal depth
For a single-element bowl, the geometric focal depth ≈ radius of curvature. A 60 mm radius means the beam converges ~60 mm from the transducer. Real transducers (e.g. NeuroFUS CTX-500) specify this in their datasheet. Convert mm → grid points by dividing by dx.
diameter = round(50e-3 / dx); % aperture diameter
Odd diameter rule
If round(50e-3/dx) gives an even number, add 1 to make it odd. k-Wave requires an odd diameter so the bowl has a well-defined central axis grid point. Check: if mod(diameter,2)==0; diameter=diameter+1; end
source.p_mask = makeBowl([Nx,Ny,Nz], bowl_pos, ...
radius, diameter, focus_pos);
makeBowl — 5 arguments
Returns an Nx×Ny×Nz binary matrix — 1 at grid points on the bowl surface, 0 everywhere else. Compare to 2D where you used makeArc([Nx,Ny], arc_pos, radius, diameter, focus_pos) — same argument pattern, just bowl instead of arc.
% ── Signal: toneBurst, same as Level 2 ─────────
f0 = 500e3; % 500 kHz
source.p = toneBurst(1/kgrid.dt, f0, 5); % 5 cycles
toneBurst — unchanged from 2D
A 5-cycle 500 kHz tone burst. Returns a 1×Nt row vector. k-Wave broadcasts this to all source points on the mask — every grid point on the bowl surface fires the same signal simultaneously (coherent, in-phase). For per-element phase delays (steering, aberration correction), you'd use a matrix — that's Level 4+.
3
Slice sensor — never record the full 3D grid
In 3D, recording everything would exhaust RAM. Use a 2D plane instead.
⚠️
The memory trap A full-grid sensor on a 64³ grid with 1000 time steps = 64×64×64×1000×4 bytes = 1 GB. At 256³ it's 64 GB. Always use a slice sensor in 3D unless you specifically need the full volume (and have a reason for the C++ runner to handle it).

% ── Method 1: axial slice (recommended) ────────
sensor.mask = zeros(Nx, Ny, Nz);
Start with an all-zero 3D matrix
This is an Nx×Ny×Nz matrix of zeros. You then set 1s wherever you want to record pressure. The sensor only stores data at active (=1) voxels, saving memory.
sensor.mask(:, :, Nz/2+1) = 1; % z mid-plane
The axial slice trick
(:, :, Nz/2+1) selects every x and y position at a fixed z. This records a 2D axial slice through the centre of the grid. After the simulation you have Nx×Ny sensor points — one time series each. Perfect for visualising the beam profile.
sensor.record = {'p_max_all', 'p_final'};
What to record
'p_max_all' — peak pressure at each sensor point over all time steps. Great for focal mapping. 'p_final' — pressure at the last time step at every grid point (needs full grid sensor or use it with your slice). Use p_max_all for FWHM measurement.
% ── After simulation, extract the slice ────────
p_slice = squeeze(sensor_data.p_max_all(:, :, Nz/2+1));
Why squeeze() instead of reshape()
In this k-Wave version, p_max_all is returned as a full Nx×Ny×Nz 3D matrix rather than a flat vector. Indexing with (:,:,Nz/2+1) extracts one z-slice but leaves a trailing size-1 dimension — the result is 128×128×1. squeeze() removes that singleton dimension giving a clean 128×128 matrix that imagesc can display.

Note: if you see a reshape() error, this is the fix — check the actual size of p_max_all with disp(size(sensor_data.p_max_all)) first.
imagesc(kgrid.y_vec*1e3, kgrid.x_vec*1e3, p_slice);
Axes in physical units (mm)
kgrid.y_vec and kgrid.x_vec are physical position vectors in metres. Multiply by 1e3 to get mm. Now your axes show real distance, not grid indices. Always label axes — an interviewer will notice missing units.
xlabel('y [mm]'); ylabel('x [mm]');
colormap("turbo"); colorbar; axis image;

input_args = {
'DataCast', 'single', % half memory
DataCast — always 'single' in 3D
'single' uses 32-bit floats instead of 64-bit doubles — halving memory. Accuracy is unaffected because the PML limits simulation accuracy to ~10⁻⁵ anyway. For GPU: use 'gpuArray-single' instead (then call gather() on the output).
'PMLInside', false, % PML outside grid
PMLInside false — use your full grid
By default (PMLInside=true), the PML absorbing layer is placed INSIDE your grid, eating 10 grid points on each face. Setting this to false places the PML outside — your Nx×Ny×Nz is fully usable domain. Recommended for 3D where every grid point counts.
'DisplayMask', source.p_mask, % show bowl
DisplayMask — visualise during simulation
Overlays the source mask on the real-time pressure plot so you can see the bowl during simulation. Set to 'off' to skip visualisation (faster). Useful for checking the transducer is placed correctly before committing to a long run.
'PlotPML', false % cleaner plot
};
sensor_data = kspaceFirstOrder3D(kgrid, medium, ...
source, sensor, input_args{:});
kspaceFirstOrder3D — identical call signature
Same four structs as every other level: kgrid, medium, source, sensor. k-Wave detects from kgrid that this is 3D and runs the 3D solver. The call is syntactically identical to kspaceFirstOrder2D — that's intentional by the k-Wave designers.
% On GPU: replace 'single' with 'gpuArray-single'
% Then: sensor_data = gather(sensor_data);
GPU path reminder
With 'DataCast','gpuArray-single', sensor_data stays on the GPU as a gpuArray. You can't index it normally. Either call gather(sensor_data.p_max_all) to pull it back, or add 'DataRecast',true as an input argument to let k-Wave gather it automatically.

% ── 1. Check geometry BEFORE running ───────────
voxelPlot(double(source.p_mask)); view([127, 18]);
Always check geometry first
Before running a simulation (which may take hours at 256³), always call voxelPlot to confirm the bowl is in the right position and facing the right direction. A bowl placed at the wrong location or facing backwards is a common mistake. Costs 1 second to check, saves hours.
% ── 2. Animate the 3D field slice by slice ─────
flyThrough(sensor_data.p_final); % steps through z
flyThrough — needs full-grid p_final
p_final is the pressure at the last time step at EVERY grid point — it requires sensor.mask = ones(Nx,Ny,Nz) or the 'p_final' record mode. flyThrough then steps through z-slices. Use this for a quick visual check; use imagesc for measurements.
% ── 3. Quantitative slice plot ──────────────────
p_xy = squeeze(sensor_data.p_final(:,:,Nz/2));
squeeze() — removes the singleton z dimension
sensor_data.p_final(:,:,Nz/2) selects one z-slice but returns an Nx×Ny×1 matrix (still 3D). squeeze() removes the size-1 dimension, giving you an Nx×Ny 2D matrix that imagesc can display. Without squeeze, imagesc shows a grey image.
imagesc(kgrid.y_vec*1e3, kgrid.x_vec*1e3, p_xy);
colormap("turbo"); axis image; colorbar;
xlabel('y [mm]'); ylabel('x [mm]'); title('XY plane at focus');

What you should see — bowl geometry check
A yellow spherical cap (the bowl surface) sitting against the left wall of the grid, centred in y and z. The bowl is slightly curved — this curvature is what focuses the beam. If your bowl looks flat, radius is too large. If it looks like a full sphere, diameter is too large.
If yours looks different

Bowl not visible: voxelPlot requires the source mask to have at least one active voxel — check makeBowl arguments.

Bowl touching the grid edge: diameter is too large for your grid size, reduce it.

Bowl in the wrong position: check bowl_pos coordinates — bowl_x = pml_size + 2 sits just clear of the PML on the left wall, Ny/2+1 and Nz/2+1 mean centred in the other two directions.

⚙ Design decisions in this simulation
128³ cubic grid, not 64³ or 220×128×128
We use a cubic grid because in real clinical practice the standard is always cubic — BRIC, PRESTUS, and K-Plan all use 256×256×256. We use 128 instead of 256 so this runs in 1–2 minutes on any laptop. The physics is identical — replace 128 with 256 in one place and you have a clinical-resolution simulation.
Bowl placed clear of the PML
The PML (10 grid points by default) absorbs waves to stop them reflecting off the edge of the grid — but it also damps anything placed inside it. We put the bowl two grid points past the PML (bowl_x = pml_size + 2) so the source radiates at full strength. The radius of curvature and the aperture diameter are both set to round(30e-3 / dx) — for a single bowl, focal depth is approximately equal to the radius of curvature, so this puts the focus 30mm in front of the bowl, and the 30mm aperture fits comfortably inside the 64mm cube.
t_end calculated from the geometry
Instead of a fixed number, t_end is calculated as the time for the wave to cross the whole grid plus the burst duration: t_end = (Nx*dx)/c0 + 5/f0. This automatically scales if you change the grid size or frequency. If t_end is too long the wave bounces off the PML, returns, and interferes with the outgoing beam, producing a stripe pattern instead of a clean focal spot.
PMLInside = true (default, not changed)
We do not set PMLInside = false. When PMLInside is false, k-Wave expands the grid by adding PML layers outside your defined dimensions. This changes the sensor array size and breaks the reshape step. Always leave PMLInside at its default (true) unless you are using the C++ binary with HDF5 output.

clearvars; close all;
%% ═══ 1. GRID ════════════════════════════════════════════
f0 = 500e3; % transducer frequency [Hz]
c0 = 1500; % water sound speed [m/s]
rho0 = 1000; % water density [kg/m^3]
dx = c0 / (6 * f0); % 6 PPW -> 0.5 mm
Nx = 128; Ny = 128; Nz = 128; % 64 mm cube
kgrid = kWaveGrid(Nx, dx, Ny, dx, Nz, dx);
disp(['Grid: ' num2str(Nx) '^3 at ' num2str(dx*1e3) 'mm = ' ...
num2str(Nx*dx*1e3) 'mm cube']);
%% ═══ 2. MEDIUM + TIME ═══════════════════════════════════
medium.sound_speed = c0;
medium.density = rho0;
% t_end = time to cross the whole grid + burst duration
t_end = (Nx * dx) / c0 + 5/f0; % 42.7 µs + 10 µs ≈ 53 µs
kgrid.makeTime(medium.sound_speed, 0.3, t_end);
%% ═══ 3. SOURCE — bowl placed clear of the PML ═══════════
% With the default PML (10 grid points), the outer 10 grid points of
% the grid ARE the absorbing layer — the bowl must sit beyond them.
pml_size = 10;
bowl_x = pml_size + 2; % x = 12, safely inside
bowl_pos = [bowl_x, Ny/2+1, Nz/2+1]; % [grid points]
% For a single bowl, focal depth is approximately the radius of curvature.
% We want the focus 30mm from the bowl, so ROC = 30mm.
radius = round(30e-3 / dx); % 60 grid points
diameter = round(30e-3 / dx); % aperture 30mm
if mod(diameter, 2) == 0
diameter = diameter + 1; % makeBowl needs odd
end
focus_pos = [bowl_x + radius, Ny/2+1, Nz/2+1]; % 30mm in front of bowl
source.p_mask = makeBowl([Nx, Ny, Nz], bowl_pos, radius, diameter, focus_pos);
% Drive at a realistic amplitude so the units are meaningful.
% (Still a transient burst — beam shape is the lesson, not ISPPA.)
source_amp = 60e3; % 60 kPa at the surface
source.p = source_amp * toneBurst(1/kgrid.dt, f0, 5);
disp(['Transducer: ROC = ' num2str(radius*dx*1e3) 'mm, aperture = ' ...
num2str(diameter*dx*1e3) 'mm, bowl at x-index ' num2str(bowl_x)]);
%% ═══ 4. SENSOR ══════════════════════════════════════════
% 'p_max_all' records the peak at every grid point and ignores
% sensor.mask entirely — the mask below only matters for records like 'p'.
sensor.mask = zeros(Nx, Ny, Nz);
sensor.mask(:, :, Nz/2+1) = 1;
sensor.record = {'p_max_all'};
%% ═══ 5. GEOMETRY CHECK ══════════════════════════════════
figure;
voxelPlot(double(source.p_mask)); view([127, 18]);
title('Bowl geometry — should be clear of all grid edges');
pause(2);
%% ═══ 6. RUN ═════════════════════════════════════════════
disp('Starting simulation...');
sensor_data = kspaceFirstOrder3D(kgrid, medium, source, sensor, ...
'DataCast', 'single', 'PlotPML', false);
disp('Simulation complete.');
%% ═══ 7. VISUALISE + REPORT ══════════════════════════════
p_slice = squeeze(sensor_data.p_max_all(:, :, Nz/2+1));
figure;
imagesc(kgrid.y_vec*1e3, kgrid.x_vec*1e3, p_slice*1e-3); % kPa
colormap("turbo"); cb = colorbar; ylabel(cb, 'Peak pressure [kPa]');
axis image;
xlabel('y [mm]'); ylabel('x [mm]');
title('Peak pressure — XY plane at z = centre');
% mark the true peak
[p_max_val, idx] = max(p_slice(:));
[row, col] = ind2sub(size(p_slice), idx);
hold on;
plot(kgrid.y_vec(col)*1e3, kgrid.x_vec(row)*1e3, 'w+', ...
'MarkerSize', 14, 'LineWidth', 2);
text(kgrid.y_vec(col)*1e3 + 2, kgrid.x_vec(row)*1e3, 'focus', ...
'Color', 'white', 'FontSize', 10, 'FontWeight', 'bold');
% focal distance measured from the bowl surface
focal_dist_mm = (row - bowl_x) * dx * 1e3;
focal_gain = p_max_val / source_amp;
disp('─────────────────────────────────────');
disp(['Peak pressure: ' num2str(p_max_val*1e-3, 3) ' kPa']);
disp(['Focal distance: ' num2str(focal_dist_mm, 3) ' mm from bowl' ...
' (expect ~ROC = 30mm, slightly less is normal)']);
disp(['Focal gain: ' num2str(focal_gain, 3) 'x source amplitude']);
disp('─────────────────────────────────────');
disp('NOTE: transient burst, on-grid bowl -> beam shape only.');
disp('Calibrated MPa / ISPPA needs CW + kWaveArray (water-tank level).');

What you should see — peak pressure map
A converging beam visible as a bright vertical column, narrowing to a hot focal spot marked with a white cross. The beam enters near the left wall (just clear of the PML, where the bowl sits) and focuses roughly 30mm further in. The focal zone is elongated axially (top to bottom in this view) and narrow laterally — this cigar shape is correct physics at 500 kHz. The colour scale is in kPa: with a 60 kPa source, the focal spot should reach well above the source amplitude — that ratio is the "focal gain" printed at the end of the script.
If yours looks different

Vertical stripes across the whole image: t_end is too long and the wave has bounced back — reduce t_end.

Empty or all blue: simulation ran but no pressure recorded — check sensor.mask is set correctly.

Focus label in wrong position: the white cross marks the true peak, which may differ slightly from focus_pos due to grid discretisation — this is normal.

Axes show much larger values than expected: PMLInside was set to false and the grid expanded — remove that line.

Check yourself
Three questions before moving to Level 4
Question 1 of 3
You change from kWaveGrid(Nx,dx,Ny,dy) to kWaveGrid(Nx,dx,Ny,dy,Nz,dz). What else do you need to change in source.p_mask?
In k-Wave, every mask (source.p_mask, sensor.mask) must match the grid dimensions exactly. A 3D kgrid requires Nx×Ny×Nz binary matrices for all masks. k-Wave does zero automatic resizing — if the dimensions don't match, you get a MATLAB dimension mismatch error.
Question 2 of 3
Your sensor.mask = ones(Nx,Ny,Nz). The simulation crashes with an out-of-memory error. What is the fastest fix?
A full-grid sensor stores a time series at every single voxel — at 64³ with 1000 time steps that's already hundreds of MB. Switching to a slice sensor (one 2D plane) reduces recorded data by a factor of Nz. This is always the first fix. If you genuinely need the full 3D field, use the C++ runner with HDF5 output.
Question 3 of 3
In a 3D simulation with skull (coming in Level 4), the actual focal spot lands 8mm away from your focus_pos. What is the correct fix?
The focal shift is caused by skull refraction — not by a simulation error. focus_pos only sets the bowl's geometric orientation. Moving bowl_pos changes transducer location entirely (wrong). Changing focus_pos just reorients the bowl (doesn't fix aberration). The correct solution is time-reversal: record pressure at focus_pos, run time-reversed back through the skull, compute per-element delays to pre-compensate. This is the full Level 4 topic.
