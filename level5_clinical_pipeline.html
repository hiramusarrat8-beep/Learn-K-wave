# SimNIBS charm segmentation (run in terminal, NOT MATLAB)
# Input: T1-weighted MRI NIfTI. Takes 30-60 min.
# Output: m2m_subjectID/ with tissue label NIfTI files
charm sub-01 sub-01_T1w.nii.gz
What charm produces
Outputs tissue label NIfTI files: WM, GM, CSF, skull, skin, air. Each voxel gets a label 0-5. PRESTUS reads these and assigns IT'IS acoustic properties. k-Wave itself never sees labels — only the final c, rho, alpha matrices.
# PETRA alternative (no CT needed)
PETRA — pseudo-CT from MRI
PETRA is a Siemens UTE MRI sequence (TE~0.07ms) that makes bone visible. The UCL pipeline converts PETRA intensity to pseudo-HU then runs the same Aubry formula. Limitation: signal dropout in thick bone. Research use only, not clinical standard.

% Load CT and T1 MRI as plain numerical matrices
ct_img = niftiread('sub-01_pct.nii'); % HU matrix
niftiread — returns raw matrix
niftiread returns a plain MATLAB numerical matrix (int16 or double). For CT, values are Hounsfield Units typically -1000 to +3000. Pass this directly into the Aubry formula — same as your synthetic HU_map from Level 4.
t1_img = niftiread('sub-01_T1w.nii'); % anatomy overlay
T1 MRI — used for visualisation, not physics
The T1 MRI is used for: (1) overlaying pressure maps onto anatomy in 3D Slicer; (2) defining target coordinates. It does NOT set acoustic properties. Brain acoustic properties come from SimNIBS tissue labels + IT'IS lookup, not MRI intensity.
header = niftiinfo('sub-01_pct.nii'); % voxel size
niftiinfo — metadata you need
header.PixelDimensions gives voxel size in mm. Real CT is typically 1x1x1mm isotropic. You need this to compute scale_factor = CT_voxel_size / sim_dx, for resampling CT to match your simulation grid.
% Resample CT to simulation grid resolution
dx_sim = 0.5e-3; % sim grid spacing [m]
scale = header.PixelDimensions / (dx_sim*1e3);
Scale factor — CT to sim resolution
CT is typically 1mm/voxel. Sim grid is 0.5mm/voxel. scale = 1/0.5 = 2. After resampling, ct_img has 2x more voxels per mm, matching your simulation grid.
scale = CT_voxel_mm / sim_dx_mm (e.g. 1.0 / 0.5 = 2.0)
ct_img = imresize3(ct_img, 'cubic', 'Scale', scale);
imresize3 — 3D resampling
Resamples the 3D CT matrix. Use 'cubic' for smooth HU interpolation. After this, ct_img has the same voxel size as your simulation grid. Pass it straight into the Aubry formula as HU_map — identical to Level 4.
% Transducer and focus positions (from 3D Slicer fiducials)
focus_coords = round([99, 161, 202] .* scale);
Coordinates from 3D Slicer
Place a fiducial marker in 3D Slicer at your brain target. Slicer gives voxel coordinates in CT image space. Multiply by scale to get sim grid indices. Add 1 if Slicer uses 0-indexing (MATLAB indexing starts from 1).
bowl_coords = round([90, 193, 262] .* scale);

% Crop 256^3 subgrid centred on midpoint between bowl and focus
midpoint = round((bowl_coords + focus_coords) / 2);
Why 256^3 centred on midpoint?
The sim grid only needs to contain the transducer, skull, and target — not the entire head. Centring on the midpoint guarantees both are inside the grid with PML margin. 256 points at 0.5mm = 128mm — large enough for most tFUS geometries.
grid_span = 256 * dx = 256 * 0.5mm = 128mm
half = 128; % 256/2 = 128 grid points each side
HU_map = ct_img(midpoint(1)-half+1:midpoint(1)+half, ...
midpoint(2)-half+1:midpoint(2)+half, ...
midpoint(3)-half+1:midpoint(3)+half);

Same code as Level 4 from here
Once you have HU_map (whether from real CT or synthetic), the Aubry conversion is identical. This is the pedagogical point: Level 4 applies directly to real patient data. The only difference at Level 5 is how HU_map was created.
porosity = 1 - (HU_map / 1000);
medium.sound_speed = 2900 - 1480 .* porosity;
medium.density = 1900 - 900 .* porosity;

% Convert grid indices to physical positions [m]
bowl_pos_m = [kgrid.x_vec(bowl_coords(1)), ...
kgrid.y_vec(bowl_coords(2)), kgrid.z_vec(bowl_coords(3))];
Grid index to physical position
kWaveArray takes positions in metres, not grid indices. kgrid.x_vec(N) converts grid index N to physical position. BRIC does exactly this conversion before calling addAnnularArray.
x_vec(N) = (N - Nx/2 - 1) * dx [metres]
focus_pos_m = [kgrid.x_vec(focus_coords(1)), ...
kgrid.y_vec(focus_coords(2)), kgrid.z_vec(focus_coords(3))];
% Create kWaveArray (off-grid, Wise et al. 2019)
kWaveArray vs makeBowl
makeBowl snaps source points to nearest grid points — staircasing error. Wise 2019 showed this error does NOT converge to zero with finer grids. kWaveArray uses a band-limiting convolution giving <0.3% amplitude error at 3 PPW. Always use kWaveArray for real transducers.
karray = kWaveArray('SinglePrecision', true);
% NeuroFUS CTX-500-4: 4-element annular array
source_roc = 63.2e-3; % radius of curvature [m]
Transducer specs from datasheet
These values are specific to the NeuroFUS CTX-500-4. For any transducer, look up: radius of curvature (focal depth), element diameters, frequency. The diameters array defines the four concentric ring boundaries.
diameters = [0, 32.2, 38.2, 54.4] * 1e-3; % [m]
karray.addAnnularArray(bowl_pos_m, source_roc, diameters, focus_pos_m);
addAnnularArray — one call for all 4 elements
Creates 4 concentric ring elements. Each ring becomes an independent source element with its own amplitude and phase. This is how CTX-500-4 achieves electronic steering and aberration correction — by setting different phases on each ring.
source.p_mask = karray.getArrayBinaryMask(kgrid);
getArrayBinaryMask — replaces makeBowl result
Returns the same thing as makeBowl — an Nx x Ny x Nz binary matrix. The difference is HOW the 1s were chosen: kWaveArray distributes source weight optimally, while makeBowl snaps to nearest. All downstream code (sensor, kspaceFirstOrder3D) stays identical.
% Per-element amplitude and phase (aberration correction)
source_amp = repmat(pressure, [1, 4]); % 1x4
Per-element amplitude and phase
source_amp is 1x4 (one per element). source_phase is 1x4 in radians. For the initial forward simulation (no correction), set all phases to 0. After time-reversal, these become the per-element corrected delays.
source_phase = deg2rad([0, 0, 0, 0]); % no correction yet
source_sig = createCWSignals(kgrid.t_array, freq, source_amp, source_phase);
createCWSignals replaces toneBurst
Creates a continuous wave sinusoidal signal for each element. Unlike toneBurst (N cycles then silence), CW runs until steady state — matching actual tFUS transducer output. Returns a 4 x Nt matrix.
source.p = karray.getDistributedSourceSignal(kgrid, source_sig);
getDistributedSourceSignal — the slow step
Distributes the 4-element signals to every grid point in source.p_mask, weighted by the off-grid band-limiting function. Returns N_gridpoints x Nt matrix. For 256^3 grids this can take 10-30 min. Run it once and save to disk if re-running.

%% Sensor: full grid — needs C++ runner
sensor.mask = ones(Nx, Ny, Nz);
Full grid sensor — only viable with C++ runner
Full grid on 256^3 recording all time steps = ~64 GB. The key trick: record only the last few periods (record_start_index), then extractAmpPhase reduces it to a single amplitude map. Final output is a 256^3 single-precision matrix ~ 64 MB.
sensor.record = {'p'};
ppp = round(ppw / cfl); % points per period
record_start_index — the memory-saving trick
CW simulation runs until steady state (~10-20 wave travel times). Only the last few periods are needed for extractAmpPhase. Setting record_start_index reduces recorded data from ~10,000 to ~300 time steps — 30x less memory.
sensor.record_start_index = kgrid.Nt - 3 * ppp + 1;
%% C++ runner via HDF5
input_file = fullfile(output_dir, 'kwave_input.h5');
output_file = fullfile(output_dir, 'kwave_output.h5');
% Step 1: save inputs to HDF5 (MATLAB writes, does not run)
kspaceFirstOrder3D(kgrid, medium, source, sensor, input_args{:}, ...
SaveToDisk — HDF5 pipeline step 1
With 'SaveToDisk', kspaceFirstOrder3D writes all inputs to an HDF5 file and exits without running the simulation. The C++ binary reads this file directly — no MATLAB memory overhead. This is the standard pattern in BRIC, PRESTUS, BabelBrain, K-Plan.
kspaceFirstOrder3D(..., 'SaveToDisk', 'kwave_input.h5')
'SaveToDisk', input_file);
% Step 2: run C++ binary
kwave_bin = fullfile(fileparts(which('kspaceFirstOrder3D')), ...
'binaries', 'kspaceFirstOrder-OMP');
Finding the C++ binary automatically
The C++ binary is in the binaries/ subfolder of the k-Wave toolbox. fileparts(which('kspaceFirstOrder3D')) finds the toolbox path automatically. BRIC's linux_system case uses the same approach. On Windows: kspaceFirstOrder-OMP.exe
system([kwave_bin ' -i ' input_file ' -o ' output_file ...
' --p_raw -s ' num2str(sensor.record_start_index) ' -t 12'], '-echo');
% Step 3: read results back into MATLAB
sensor_data.p = h5read(output_file, '/p');
h5read — read HDF5 dataset
The C++ binary writes results to /p in the output HDF5. h5read loads this back into MATLAB. Shape: N_sensor_points x N_recorded_timesteps. Much smaller than full time series.
% Step 4: extract amplitude at source frequency
p_amp = extractAmpPhase(sensor_data.p, 1/kgrid.dt, freq, ...
extractAmpPhase — FFT at source frequency
For CW simulation, sensor_data.p is a steady-state sinusoid at each point. extractAmpPhase performs an FFT and returns the amplitude at exactly freq. This eliminates transients and harmonics. Use this instead of p_max_all for CW.
'Dim', 2, 'Window', 'Rectangular');
p_amp = reshape(p_amp, Nx, Ny, Nz);

0.5 x p_max = -6dB in pressure amplitude. 0.71 x p_max = -3dB in pressure = FWHM (half-max intensity). BRIC uses 0.71 but incorrectly labels it -6dB. Use 0.5 for correct -6dB reporting.

p_out = imresize3(p_out, 'cubic', 'Scale', 1./scale);

The output pressure_field.nii can be loaded in 3D Slicer and overlaid on the T1 MRI to visualise where the focus landed in the patient's brain. niftiwrite uses the original CT header to preserve voxel size and orientation.

Three questions covering Level 5 concepts
Question 1 of 3
Why does the CW simulation use extractAmpPhase() to get pressure, while the Level 3 toneBurst simulation used p_max_all?
A toneBurst simulation has a clear pressure peak — p_max_all captures it correctly. A CW simulation records a steady-state sinusoid at each sensor point. The peak of this sinusoid is the amplitude you want, but p_max_all could be slightly inflated by transient startup effects. extractAmpPhase does an FFT on the steady-state window and returns the amplitude at exactly freq, eliminating transients and harmonic content.
Question 2 of 3
BRIC uses threshold p > 0.71 * max_pressure and labels it "−6dB focal volume." What is actually being computed?
0.71 x p_max = (1/√2) x p_max. Since intensity is proportional to p², at this pressure the intensity is 0.5 x I_max — the half-maximum intensity, i.e. FWHM, i.e. -3dB in intensity. The true -6dB pressure threshold (pressure at which intensity is 0.25 x I_max) would use 0.5 x p_max. BRIC's code computes the correct clinical FWHM but uses an incorrect comment. This distinction matters for reporting — always state which definition you are using.
Question 3 of 3
A colleague says "we don't need k-Wave knowledge since we just use PRESTUS — it's all automated." What is the most important reason this view is wrong?
Automated tools abstract complexity — they don't remove it. When PRESTUS produces a focal spot 12mm from your target, you need to distinguish: skull refraction (expected, apply aberration correction), CT-to-grid alignment error (a pipeline bug), or wrong transducer geometry (wrong datasheet values). Without understanding PPW, CFL, Aubry, kWaveArray, and extractAmpPhase, you cannot make that distinction. Every tFUS lab expects its members to understand the simulation, not just operate the tool.
